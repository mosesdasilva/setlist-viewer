01 · Big picture

What is this repo?

Setlist Viewer is a static website for musicians. It shows one song at a time, including song details, Nashville Number Chart bars, melody notes, lyrics, and section navigation.

HTML

The skeleton

Names the header, buttons, dialog, and empty area where songs appear.

CSS

The appearance

Controls color, spacing, mobile layout, columns, themes, and print behavior.

JavaScript

The behavior

Creates songs on screen, reacts to clicks, and remembers preferences.

“Static” does not mean “non-interactive.” It means no application server or database is needed. The browser receives files and runs the JavaScript locally.

The most important mental model

03 · Run it

No installation needed

There is no framework, package manager, npm install, or compilation step required to view the committed site.

Fastest: open a file

From the repository folder on macOS:

open src/index.html

This opens the canonical split viewer. It works through file:// because data loads from a normal script file, not from a network request.

Portable: open the all-in-one file

open setlist-viewer-portable.html

Use this when sharing or testing one self-contained file. Do not edit it directly.

Optional: local web server

python3 -m http.server 8000 --directory src

Keep that Terminal window open, then visit http://localhost:8000/. Stop the server with Control+C.

Which run method should I use? Start with open src/index.html. Use the server when testing behavior closer to GitHub Pages. Use the portable file when testing the handoff artifact.
04 · Browser control flow

What happens when the page opens?

“Control flow” means the order in which code runs and decisions happen. The browser begins at src/index.html.

1. Browser reads index.htmlCreates the header, controls, picker, and empty song list.
→
2. Browser loads styles.cssApplies layout, colors, responsive rules, themes, and modes.
→
3. Browser loads chart-data.jsCreates window.SETLIST_SONGS, the runtime catalog.
→
4. Browser runs script.jsBuilds song elements, binds events, restores preferences, activates song 1.

The two runtime entry points

Main

src/index.html

Canonical split website. GitHub Pages publishes this folder.

Portable

setlist-viewer-portable.html

Same CSS, data, and JavaScript inserted into one generated file.

How a button click travels

User clicksNext Song
→
Event handler runssetActiveSong(...)
→
Page updatesactive class, header, Song Map, scroll

Other handlers apply theme, palette, columns, display mode, numbering, band size, Song Picker search, keyboard arrows, and the D theme shortcut. Preferences are stored in the browser’s localStorage.

05 · Build flow

When “no build step” still has a build command

The website needs no build to run. The repository does have a small Python generator to validate chart sources and refresh committed output files.

Canonical inputscharts/ + src/song-data.js + split UI
→
Python buildvalidate, order, combine, inline
→
Generated outputschart-data.js + portable HTML

Generate outputs

python3 tools/build.py

This validates every .chart, combines complete Charts with Legacy Songs in catalog order, then atomically replaces both generated files.

Check without writing

python3 tools/build.py --check

This fails if chart input is invalid or generated files are missing/stale. It does not repair anything. GitHub runs this exact check before deployment.

06 · Editing guide

Start from the change you want

Content

Edit a Legacy Song

Edit its object in src/song-data.js. Keep valid JSON-style punctuation. Run the build afterward.

Content

Edit a complete Chart

Edit charts/mas.chart or another .chart. Run the build; validation reports line and column errors.

Order

Reorder songs

Change ID order in charts/catalog.json. Every available song must appear exactly once. Run the build.

Appearance

Change visuals

Edit src/styles.css. Search for the class shown in HTML or created by JavaScript. Rebuild portable output afterward.

Structure

Add a visible control

Edit src/index.html. Usually add matching behavior in src/script.js and styles in src/styles.css.

Behavior

Change interactions

Edit src/script.js. Follow the pattern: find element → bind event → update DOM/state → save preference if needed.

A safe beginner loop

  1. Run git status --short. Notice existing work; do not overwrite it.
  2. Open the current viewer so you know the “before” state.
  3. Make one small change in a canonical source file.
  4. Run python3 tools/build.py if any generated output depends on it.
  5. Refresh both split and portable viewers. Check desktop and narrow mobile width.
  6. Run automated checks. Review git diff before committing.
Common beginner trap Editing setlist-viewer-portable.html may look successful today, but the next build erases the change. Edit split source and regenerate.
07 · Verification

Prove the change works

Run every automated test

python3 -m unittest discover -s tests -p 'test_*.py'

The tests cover chart validation, generation, Legacy Songs, rendering contracts, and the GitHub Pages workflow. They use Python’s standard library; no packages need installing.

Check generated files

python3 tools/build.py --check

Inspect your changes

git status --short
git diff

Manual Done Criteria

  • src/index.html opens and song navigation works.
  • setlist-viewer-portable.html opens and behaves the same.
  • Theme toggle and display controls still work.
  • Desktop and mobile layouts remain readable.
  • Project/session docs are updated if a decision or next step changed.
08 · Deployment

How code reaches GitHub Pages

The deployment entry point is .github/workflows/deploy-pages.yml. It runs when main is pushed, or when manually started in GitHub Actions.

1. Push mainGitHub receives the committed files.
→
2. ValidateSet up Python 3.9 → run all tests → run build --check.
→
3. Upload src/Only after validation succeeds.
→
4. Deploy PagesGitHub publishes the static website.

If tests or drift checks fail, deployment stops. This protects the live viewer from invalid charts and forgotten generated updates.

09 · Glossary

Words you will see

Repository
A project folder whose history Git tracks.
Entry point
The file or function where a process starts.
Runtime
What happens while the browser is executing the site.
DOM
The browser’s in-memory tree of HTML elements. JavaScript changes it.
Event
A click, key press, input, resize, or other occurrence code can react to.
State
Current information, such as active song, theme, or selected mode.
Canonical
The official source to edit—the source of truth.
Generated
Output created from canonical source by a command.
Build
Here: validate sources and generate browser data plus portable HTML.
Test
Code that automatically checks expected behavior or structure.
CI
Automated checks GitHub runs after code is pushed.
Deploy
Publish checked files so people can visit the website.
Legacy Song
Metadata and section summary without full chart rows.
Chart
A complete validated Nashville Number Chart from a .chart file.
